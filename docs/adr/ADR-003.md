# ADR-003: Plataforma de Despliegue

**Estado:** Aceptado  
**Fecha:** 2026-04-29  
**Decisor:** Equipo de Desarrollo  

## Contexto

La aplicación requiere una plataforma de despliegue en la nube que:
- Permita despliegue automático desde GitHub
- Sea accesible públicamente
- Tenga base de datos integrada
- Sea económica para propósitos educativos/pequeños proyectos

## Opciones Consideradas

1. **Heroku** - PaaS tradicional con Procfile, git push deploy
2. **Render** - Plataforma moderna, similar a Heroku pero más económica
3. **AWS EC2** - IaaS que requiere configuración manual
4. **Google Cloud Run** - Serverless, pay-per-use
5. **PythonAnywhere** - Hosting específico de Python, compatible con Flask

## Decisión

**Usar Render como plataforma de despliegue para el backend/web y Vercel para la app móvil.**

## Rationale

1. **GitHub Integration:** Render y Vercel soportan despliegue automático desde el repositorio.
2. **Coste:** Render ofrece un plan económico para servicios web y Vercel es ideal para frontend estático/spa.
3. **Separación de responsabilidades:** Backend y web en Flask corren en Render; la app móvil en Ionic React se despliega en Vercel.
4. **Persistencia:** Para producción se recomienda PostgreSQL, aunque el MVP funciona con SQLite en la app actual.
5. **Escalabilidad:** Render soporta servicios Flask con Procfile y configuración similar a Heroku.

## Proceso de Despliegue

Render ejecuta deploy automático cuando se hace push a la rama principal del repositorio.

Para el backend Flask:
```bash
# Asegurar que el código está en main
git push origin main
```

En Render, la aplicación se configura como servicio web con el comando:
```bash
gunicorn app:app
```

Para el frontend móvil en Vercel, el repo también está conectado y despliega automáticamente cuando se hace push.

## Configuración Requerida

**Procfile:**
```
web: gunicorn app:app
```

**Variables de entorno en Render:**
- `DATABASE_URL` (si se usa PostgreSQL o un servicio gestionado)
- `FLASK_ENV=production`

**Variables de entorno en Vercel:**
- `VITE_API_BASE_URL` apuntando al backend de Render, por ejemplo `https://<your-render-service>.onrender.com/api`

## Consecuencias

**Positivas:**
- Arquitectura real: backend Flask en Render y frontend móvil en Vercel.
- Deploy automático desde GitHub para ambos servicios.
- Separación clara entre API y app móvil.

**Negativas:**
- Requiere coordinar dos despliegues distintos.
- Si Render no está actualizado, el frontend en Vercel puede consumir una versión antigua.

**Nota sobre SQLite en Heroku:**
- Heroku tiene filesystem efímero (se reescribe cada deploy)
- Para producción real, usar PostgreSQL de Heroku
- Para desarrollo/demo, SQLite es suficiente

## Consecuencias

**Positivas:**
- Integración nativa con GitHub
- Despliegue de una línea: `git push heroku main`
- Soporte 24/7 y documentación excelente
- Facilidad de entendimiento para estudiantes

**Negativas:**
- Costo puede aumentar con uso intensivo
- Dyno "free" ya no disponible (desde Nov 2022)
- Lock-in a plataforma Heroku (aunque Procfile es estándar)

## Alternativas Recomendadas (2026)

Dado que Heroku cambió su modelo de precios:
- **Render:** $0.10/hour por dyno (más económico)
- **Railway:** Excelente UX, planes acordes a uso
- **Fly.io:** Rendimiento global, pricing competitivo

Todos usan estándares similares (Dockerfile, etc.) facilitando migración futura.
