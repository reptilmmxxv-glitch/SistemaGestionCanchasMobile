# ADR-002: Servidor WSGI

**Estado:** Aceptado  
**Fecha:** 2026-04-29  
**Decisor:** Equipo de Desarrollo  

## Contexto

Para el despliegue en producción, se requiere un servidor WSGI que:
- Maneje múltiples conexiones concurrentes
- Sea compatible con plataformas en la nube (Heroku, Render, etc.)
- Tenga bajo overhead de recursos
- Permita debugging eficiente

## Opciones Consideradas

1. **Gunicorn** (Green Unicorn) - Servidor WSGI HTTP puro
2. **Waitress** - Servidor WSGI puro en Python
3. **uWSGI** - Servidor ultra rápido pero complejo
4. **Flask development server** - Solo para desarrollo

## Decisión

**Usar Gunicorn como servidor WSGI en producción.**

## Rationale

1. **Compatibilidad:** Gunicorn es el estándar de facto en plataformas cloud (Heroku, Render, AWS)
2. **Rendimiento:** Workers asincronos (Gevent) pueden manejar miles de conexiones
3. **Simplicidad:** Configuración simple mediante línea de comandos
4. **Portabilidad:** Los mismos comandos funcionan en Windows, Linux y macOS
5. **Seguridad:** No expone información sensible de la aplicación

## Diferencia: Debug vs Producción

### Desarrollo
```bash
flask run  # Puerto 5000, reload automático, debug=True
```

### Producción
```bash
gunicorn app:app --workers=4 --worker-class=sync --bind=0.0.0.0:8000
```

**Diferencias clave:**
- **Debug mode deshabilitado:** No muestra stack traces internos
- **Workers múltiples:** Gunicorn gestiona 4+ procesos paralelos
- **Worker class:** Seleccionado según la carga esperada
- **Binding:** Escucha en todas las interfaces (0.0.0.0) en puerto 8000

## Consecuencias

**Positivas:**
- Aplicación preparada para producción
- Manejo seguro de múltiples solicitudes simultáneas
- Compatible con reverse proxies (Nginx, Apache)
- Fácil de escalar horizontalmente

**Negativas:**
- Curva de aprendizaje para configuración avanzada
- Requiere monitoreo en producción
- Necesita coordinación con load balancers
